Translation Strategy

The original v1 plan called for an OPUS-MT input-translation stage to convert non-English documents to English before NLP. That stage was cut. It was a 2023 design constraint that no longer applies in 2026: modern frontier-class small models (we use gpt-4o-mini) are natively multilingual — they read Chinese, Arabic, French, Spanish, Portuguese, and CJK scripts directly and emit English output in a single call. Adding OPUS-MT in front of them would mean translating twice (lossy MT, then LLM re-comprehension), shipping 400 MB+ of model weights, and adding a failure mode without measurable quality gain.

Lane Plan said Built as (today) Why
Input translation
(source → English for NLP)		 OPUS-MT (offline, free) Removed. LLM consumes source language directly. gpt-4o-mini is natively multilingual. Verified on gov.cn (Chinese) → English summary + entities in one shot.
Canonical analysis layer
(score / topic / entities / obligations)		 English-only (post-OPUS-MT) Generated in English regardless of source language. summary, compliance_summary, obligation.action all stored in English. Single canonical form simplifies search, matching, and dedup. Source-language raw text is preserved on raw_documents / source_versions for audit / dashboard "view original".
Output translation
(English → client preferred lang)		 DeepL or LLM, lazy + cached M5 — pending. Translation cache schema designed, not yet implemented. Translates only short English summaries + obligation actions, never the full document. Cache key = sha256(text)+target_lang so each unique alert is paid for once per language regardless of recipient count.

When OPUS-MT comes back: if a client requires an air-gapped deployment (no external LLM) we'd swap in a local pipeline of OPUS-MT (translation) + a local LLaMA-class model (analysis). The interface (app.services.significance.score_event) is unchanged; only the backing model changes. This is a configuration switch, not a re-architecture.
